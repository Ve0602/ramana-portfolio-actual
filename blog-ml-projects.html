I Built 4 ML Projects From Scratch.
Here's What Each One Taught Me.

Fraud detection, cardiac risk AI, forest fire prediction, and a home food marketplace — every project broke something different in my thinking, then rebuilt it better. The real lessons weren't in the models.

RV

There's a version of machine learning education that looks like this: complete a Coursera course, copy a Jupyter notebook from Kaggle, submit it as a project, put it on your LinkedIn. I've seen hundreds of portfolios built this way. They're invisible to anyone hiring for real AI roles.

The version I chose was harder and slower. I built everything from scratch — the data pipeline, the model, the API, the deployment. I broke things, fixed them, broke them again. And I learned things you simply cannot learn from a course.

Here's what each project actually taught me.

Project 1 — Guardian of Your Money 🛡

🛡
Fraud Detection with Explainability
Python · XGBoost · RandomForest · Flask · IBM Cloud

This started as "I want to build a fraud detector." It ended up teaching me that the model is the least interesting part of a fraud detection system.

The interesting parts are: feature engineering that captures human deception patterns, class imbalance handling when only 0.13% of transactions are fraud, and most importantly — explainability. A model that says "FRAUD: 94%" is useless to a human investigator. A system that says "Account drained in single transaction, destination balance was zero, high-value TRANSFER at 3am" is actionable.

I built a rule-based explainability layer on top of the ensemble model. Every prediction generates 2–4 human-readable reasons drawn from 11 fraud pattern categories. That layer took longer to build than the ML model itself.

The Real Lesson

In production AI, the explanation of a decision is often more valuable than the decision itself. Models get retrained. Explanations build trust with the humans who have to act on them.

The 6.3 million training records from the PaySim dataset also taught me something about scale: what works at 10,000 records fails at 6 million unless you've thought carefully about memory, chunking, and vectorised operations. My first training run crashed the kernel. The second one ran for four hours. The third one was clean.

Project 2 — For Your Good Heart ❤

❤
Cardiac Risk Assessment — SVM Classifier
Python · SVM · scikit-learn · Flask

I chose this project because I wanted to work on something with real human stakes. Not to be dramatic — but writing a fraud detector feels abstract. Building something that could theoretically help someone understand their cardiac risk felt different.

The dataset is small by ML standards: 299 patients from the Faisalabad Institute of Cardiology. 299 records. That constraint forced me to understand something I'd been avoiding: the entire science of making reliable predictions from small data.

SVM with an RBF kernel is the right choice here — not because it's the most powerful algorithm, but because it generalises well with small datasets and doesn't overfit the way tree-based methods can. GridSearchCV over C and gamma took hours but was essential. The difference between wrong hyperparameters and right ones was the difference between a model that memorised the training data and one that actually learned cardiac risk patterns.

"The algorithm is a choice, not a default. Choosing SVM over XGBoost for 299 records is a deliberate decision grounded in bias-variance theory — not a guess."

The advice engine was the part I'm most proud of. Risk score → clinical advice tier → personalised recommendations grounded in WHO cardiovascular guidelines. I spent two evenings just reading those guidelines to make sure the advice was medically defensible, not just plausible-sounding.

The Real Lesson

Algorithm selection is domain knowledge, not preference. Small data demands different choices than big data. Knowing why you chose something is more important than knowing how to implement it.

Project 3 — Before the Forest Burns 🔥

🔥
Forest Fire Risk Intelligence — Canadian FWI
Python · FWI Algorithm · GeoJSON · Flask

This one was the most technically demanding, and the most humbling.

The Canadian Forest Weather Index system isn't a machine learning model. It's a deterministic algorithm developed over decades by fire scientists — six interconnected indices (FFMC, DMC, DC, ISI, BUI, FWI) that together describe fire behaviour from four raw weather inputs. Implementing it correctly from the original mathematical specification required reading forestry research papers in a way that no ML tutorial ever prepared me for.

# The FWI system has real mathematical teeth # FFMC = Fine Fuel Moisture Code (dead fine fuels) # DMC = Duff Moisture Code (loosely compacted forest floor) # DC = Drought Code (deep, compact organic layers) # ISI = Initial Spread Index (fire spread rate) # BUI = Buildup Index (total fuel available) # FWI = Final fire weather index combining all above def calc_fwi(isi, bui): if bui <= 80: fd = 0.626 * (bui ** 0.809) + 2 else: fd = 1000 / (25 + 108.64 * math.exp(-0.023 * bui)) return math.exp(2.72 * (0.434 * math.log(isi * fd)) ** 0.647)

Getting this right took much longer than any ML model I've trained. But it taught me something invaluable: not every problem in AI needs a neural network. Sometimes the domain experts who spent 40 years studying wildfire behaviour have already built the best model. Your job is to implement it faithfully and deploy it usefully.

The Real Lesson

Domain expertise embedded in a deterministic algorithm can outperform a neural network trained on limited data. Reading primary sources — the actual scientific papers — is a skill that distinguishes senior AI engineers from people who only know how to call model.fit().

Project 4 — MANA: Taste of Home 🍱

🍱
Home Food Marketplace — Full Stack + AI Matching
React · Spring Boot · MongoDB · SQL · Figma

The first three projects are pure ML. MANA is where I had to become a product engineer.

The AI component — dietary matching using cosine similarity on nutritional vectors — took about two days to build. The product itself took weeks. Designing user flows in Figma, building React components, wiring a Spring Boot REST API, managing two databases (MongoDB for flexible documents, PostgreSQL for billing), implementing JWT authentication — each piece was a discipline unto itself.

What MANA taught me is that AI features are only as useful as the product they live in. The matching algorithm is useless without the cook profile UI. The cook profile is useless without the order flow. The order flow is useless without the subscription management. Everything is connected, and the person who builds it all learns things that the person who only builds the model never does.

The Real Lesson

The best AI engineers can go from model to product. Understanding frontend, backend, and data layers makes your AI work ship-able. Models that never ship don't help anyone.

What All Four Projects Have In Common

Looking back across all of them, I see three patterns that appeared in every project without exception:

  • The real work starts after the model trains. Deployment, explainability, edge cases, user experience — these take longer than training and matter more.
  • The hardest bugs are conceptual, not technical. Wrong assumption about what the data means. Wrong choice of evaluation metric. Wrong mental model of the problem. These can't be fixed by Googling.
  • Shipping something live changes your understanding fundamentally. A project that runs in a Jupyter notebook is a demonstration. A project that runs on IBM Cloud with a real API endpoint is a product. The gap between those two things is enormous and worth crossing.

If you're building your portfolio now: Don't stop at the model. Build the API. Deploy it somewhere. Make it accessible to a URL. The difference between a Kaggle notebook and a deployed Flask app on IBM Cloud is the difference between "I studied ML" and "I build AI systems."
